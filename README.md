# IPS S2026 RGB-D Sorting System

This repository contains the live RGB-D conveyor-belt pipeline used for:

- live object detection and segmentation
- data collection with RGB, depth, masks, labels, point clouds, and metadata
- offline dataset preparation and verification

The detailed user guide is split across the `docs/` folder. Start with `docs/README.md`.

## Quick Start

1. Install the requirements for Python and the Orbbec SDK.
2. Make sure the `pyorbbecsdk` Python bindings are available.
3. The data-collection UI also depends on the shared `offline_case/` masking utilities from the full CPS workspace.
4. From the `rgbd/` folder, run:

   ```bash
   make run
   ```

   to open the data-collection UI, or:

   ```bash
   make demo
   ```

   to launch the live inference demo.

## What Lives Where

- `rgbd/main.py` - data collection UI
- `rgbd/live/live_inference.py` - live demo UI
- `rgbd/processing/` - masking, segmentation, and debug helpers
- `rgbd/camera/` - Orbbec camera interface code
- `rgbd/data/` - saved data generated by the collection UI
- `rgbd/logs/` - runtime logs for the live demo
- `docs/` - separate guides for installation, UI controls, data layout, and models

## Data Collection Output

When a frame is saved, the collection UI writes the package into `rgbd/data/` using this layout:

```text
rgbd/data/
  images/          # cropped RGB frames
  uncropped_rgb/   # raw full-frame RGB images
  depth/           # depth visualizations saved alongside each frame
  masks/           # binary masks
  labels/          # YOLO polygon labels
  info/            # per-sample metadata text files
  pointcloud/      # saved .ply point clouds
  debug/
    live_capture_debug.log
    live_object_detection_log_YYYYMMDD_HHMMSS.log
    mask_debug/
      *_raw_mask.png
      *_raw_overlay.png
      *_post_close_mask.png
      *_post_close_overlay.png
      *_final_mask.png
      *_debug_sheet.png
```

The save logic is atomic: if any required artifact is missing, the snapshot is not committed.

## Model Files

The live demo ships with model switching built in. The current model names are documented in `docs/models.md`.

If you publish final weights to GitHub Releases, a cleaner naming scheme is:

- `best_single.pt`
- `best_multiple.pt`
- `best_multiple_depth.pt`

Keep the repo's current model map in sync with any renamed release assets.
